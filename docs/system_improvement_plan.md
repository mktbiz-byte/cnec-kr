# CNEC 인증 시스템 및 메인 페이지 개선 계획

## 1. 개요

본 문서는 CNEC 웹사이트(cnec.jp 및 cnec-kr)의 인증 시스템을 개선하고, 메인 페이지의 콘텐츠를 보강하기 위한 계획을 기술합니다. 주요 목표는 다음과 같습니다.

- 관리자와 기업 사용자의 로그인 시스템을 명확하게 분리합니다.
- 기존 사업자등록번호 기반 로그인을 이메일 기반 로그인으로 전환합니다.
- 하나의 사업자등록번호로 여러 브랜드를 등록할 수 있도록 지원합니다.
- cnecbiz.com의 메인 페이지에 회사 소개 내용을 추가하여 사이트의 정체성을 강화합니다.
- 모바일 접속 시 발생하는 SSL 인증서 문제를 해결합니다.

## 2. 현황 분석

현재 `cnec-kr`과 `cnecjp` 두 개의 프로젝트가 존재하며, 각각 다른 특징을 가지고 있습니다.

- **cnec-kr**: 최신 기술 스택을 기반으로 구축되었으며, 관리자와 기업 사용자의 역할 분리 및 인증 로직이 비교적 명확하게 구현되어 있습니다. 하지만 메인 페이지 콘텐츠가 부족합니다.
- **cnecjp**: 기존에 운영되던 일본어 버전으로, 기능은 풍부하지만 코드 구조가 복잡하고 인증 시스템이 분편화되어 있습니다. 메인 페이지는 크리에이터 중심의 캠페인 정보만 제공하고 있어 회사 소개 내용이 부족합니다.

사용자는 `cnecbiz.com`의 메인 페이지가 회사 소개 내용을 담도록 업데이트할 것을 추가로 요청했습니다.

## 3. 개선 계획

`cnec-kr`의 장점을 `cnecjp`에 통합하고, 사용자의 추가 요청을 반영하여 다음과 같이 개선 작업을 진행합니다.

### 3.1. 인증 시스템 개편 (cnec-kr 모델 적용)

#### 3.1.1. 로그인 시스템 분리

- **기업 사용자 로그인**: `/login` 경로에서 이메일과 비밀번호를 사용하여 로그인하도록 구현합니다.
- **관리자 로그인**: `/admin/login` 경로를 별도로 두어 관리자 계정만 접근할 수 있도록 제한합니다.

#### 3.1.2. 이메일 기반 로그인 전환

- 사업자등록번호 대신 이메일을 주요 식별자로 사용하도록 `AuthContext` 및 관련 컴포넌트를 수정합니다.
- 이를 통해 하나의 사업자등록번호로 여러 이메일 계정을 생성하여 다중 브랜드 지원의 기반을 마련합니다.

#### 3.1.3. 데이터베이스 스키마 수정

- `corporate_accounts` 테이블의 `business_registration_number` 컬럼에서 `UNIQUE` 제약 조건을 제거합니다.
- `email` 컬럼을 추가하고, `UNIQUE` 제약 조건을 설정하여 로그인 시 주요 키로 사용합니다.
- `is_admin` 컬럼을 활용하여 관리자와 일반 기업 사용자를 명확하게 구분합니다.

### 3.2. 메인 페이지 콘텐츠 업데이트

- `cnecjp` 프로젝트에 새로운 `HomePage.jsx` 컴포넌트를 생성합니다.
- 해당 컴포넌트에 CNEC의 회사 소개, 비전, 주요 서비스, 연락처 등 `cnecbiz.com`에 필요한 정보를 담습니다.
- `App.jsx`의 라우팅 설정을 변경하여 루트 경로(`/`)가 새로운 `HomePage.jsx`를 렌더링하도록 수정합니다.
- 기존 크리에이터 대상의 메인 페이지는 `/creators`와 같은 별도의 경로로 이전합니다.

### 3.3. SSL 인증서 문제 해결

- Netlify에 연결된 `cnec.jp` 도메인의 DNS 설정 및 SSL 인증서 구성을 검토하고, 모바일 환경에서도 HTTPS 접속이 원활하게 이루어지도록 문제를 해결합니다.

### 3.4. 기존 계정 마이그레이션

- 기존 `corporate_accounts` 테이블의 데이터를 새로운 스키마에 맞게 마이그레이션하는 스크립트를 작성합니다.
- 스크립트는 각 계정에 대해 고유한 이메일 주소를 생성하고, 사용자에게 변경된 로그인 정보를 안내하는 프로세스를 포함해야 합니다.

## 4. 기대 효과

- **보안 강화**: 관리자와 일반 사용자의 접근 경로를 분리하여 시스템 보안을 강화합니다.
- **사용자 편의성 증대**: 이메일 기반 로그인을 통해 사용자가 더 쉽게 서비스를 이용할 수 있습니다.
- **확장성 확보**: 다중 브랜드 지원을 통해 비즈니스 요구사항 변화에 유연하게 대응할 수 있습니다.
- **정보 전달 강화**: 회사 소개 페이지를 통해 방문자에게 CNEC의 가치를 명확하게 전달합니다.

## 6. 캠페인 및 결제 시스템 구축

사용자의 추가 요청에 따라, 기존 인증 시스템 개선과 더불어 캠페인 생성, 신청, 결제 및 매출 확인 기능을 구축합니다.

### 6.1. 데이터베이스 스키마 확장

캠페인과 결제 정보를 관리하기 위해 다음과 같은 테이블을 추가 또는 수정합니다.

- **`campaigns`**: 캠페인 정보를 저장합니다. (기업 ID, 캠페인명, 내용, 보상, 상태 등)
- **`campaign_applications`**: 크리에이터의 캠페인 신청 정보를 저장합니다. (캠페인 ID, 크리에이터 ID, 신청 상태 등)
- **`payments`**: 기업의 캠페인 비용 결제 정보를 저장합니다. (캠페인 ID, 기업 ID, 결제 금액, 결제 상태, 결제 수단 등)
- **`transactions`**: 시스템 내 모든 금전적 거래 내역을 기록합니다. (결제, 보상 지급 등)

### 6.2. 캠페인 관리 기능

- **기업 사용자**: 
    - 새로운 캠페인을 생성하고 비용을 결제할 수 있는 인터페이스를 제공합니다. (`/corporate/campaigns/create`)
    - 자신이 생성한 캠페인 목록과 상태를 확인합니다. (`/corporate/campaigns`)
    - 캠페인에 신청한 크리에이터 목록을 확인하고 참여를 승인/거절합니다.
- **크리에이터 사용자**:
    - 진행 중인 캠페인 목록을 확인하고 상세 정보를 조회합니다. (`/campaigns`)
    - 원하는 캠페인에 신청합니다.
- **관리자**:
    - 생성된 모든 캠페인을 관리하고 상태를 변경할 수 있습니다.
    - 캠페인 신청 현황을 모니터링합니다.

### 6.3. 결제 시스템 구축

- 일본 내에서 사용 가능한 결제 게이트웨이(예: Stripe, GMO Payment Gateway 등)를 연동하여 기업이 캠페인 비용을 결제할 수 있도록 합니다.
- 결제 성공 시 `payments` 테이블에 관련 정보가 기록되고, 해당 캠페인이 활성화됩니다.
- 크리에이터에 대한 보상 지급은 기존의 출금 시스템을 활용하되, `transactions` 테이블에 기록하여 자금 흐름을 명확히 합니다.

### 6.4. 매출 확인 기능 (관리자 페이지)

- 관리자 대시보드에 매출 확인 섹션을 추가합니다.
- 기간별, 캠페인별, 기업별 매출 데이터를 필터링하고 조회할 수 있는 기능을 제공합니다.
- 관련 데이터를 시각화(차트, 그래프)하여 직관적인 분석을 지원합니다.
